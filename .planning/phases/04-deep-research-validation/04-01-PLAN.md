Create research-behaviors.md following the same structural pattern as skill-behaviors.md (numbered sections with directive headings, professional tone, ends with italicized file purpose note).

The file must contain these sections:

**Section 1: Purpose and Loading** (~10 lines)
- State that this file is loaded for every /privacy query alongside skill-behaviors.md
- Clarify role: teaches HOW to perform live research (complements skill-behaviors.md which teaches HOW to format responses)
- Does NOT contain substantive privacy knowledge

**Section 2: WebFetch Source Compatibility** (~25 lines)
- Embed the verified compatibility matrix as a table:
  - ACCESSIBLE: FPF (fpf.org/blog/), IAPP (iapp.org/resources/article/), EDPB (edpb.europa.eu with _en suffix), ICO (ico.org.uk/for-organisations/), CPPA (cppa.ca.gov)
  - BLOCKED: FTC (403 block), EPIC (timeout), CDT (Cloudflare block)
- Instruction: NEVER attempt WebFetch on blocked sources. Use static knowledge for these and note in degradation header if they were relevant to the query.
- Include the "Best URL Pattern" column for each accessible source so Claude knows what URL shapes to construct

**Section 3: Source Selection** (~30 lines)
- From the sources matched by SKILL.md routing (Steps 2-4), select up to 4 for live fetching
- Priority order:
  1. FPF first (when relevant): fetch fpf.org/blog/ to check for recent FPF analysis
  2. Primary regulatory source: if query involves a specific regulation, fetch that regulator's site (EDPB for GDPR, CPPA for CCPA, ICO for UK GDPR)
  3. IAPP: for broad privacy topics, IAPP resource articles provide cross-cutting coverage
  4. Secondary source: choose based on topic relevance from remaining accessible sources
- Only select sources that are both (a) relevant to the query topic and (b) listed as ACCESSIBLE in the compatibility matrix
- If fewer than 3 sources are relevant, fetch fewer -- do not pad with irrelevant sources

**Section 4: URL Construction** (~30 lines)
- Instruct Claude to read the loaded nav guide(s) for each selected source and use their URL patterns to construct the most targeted URL for the specific query
- Key principle: NEVER fetch homepages or generic index pages. Always construct a specific content URL
- FPF special case: fetch fpf.org/blog/ first (the blog index is confirmed accessible and shows recent post titles). If a relevant title is found, fetch that specific blog post URL for full content. Do NOT attempt fpf.org/issue/ URLs (confirmed 404 via WebFetch)
- IAPP: use iapp.org/resources/article/[slug]/ pattern for known resource articles (e.g., us-state-privacy-legislation-tracker)
- EDPB: use pages with _en suffix
- ICO: use ico.org.uk/for-organisations/[topic]/ pattern
- CPPA: direct page URLs
- Include 2-3 examples of targeted URL construction for different query types (children's privacy, GDPR enforcement, AI regulation)

**Section 5: WebFetch Execution** (~25 lines)
- Instruct Claude to call WebFetch with:
  - URL: the targeted content URL from Section 4
  - Prompt: a focused question tied to the user's query, NOT a generic "summarize this page"
- Provide 3-4 example prompts showing the pattern:
  - "What recent developments, enforcement actions, or regulatory updates related to [user's topic] are described on this page?"
  - "What guidance or requirements related to [user's topic] does this page describe?"
  - "List any publications, reports, or analysis related to [user's topic] with their titles, dates, and key findings."
- Note: WebFetch uses a small model (Haiku) to process content, so specific prompts get better results than vague ones
- Sequential execution assumed -- fetch sources one at a time

**Section 6: Content Integration** (~30 lines)
- Static knowledge from reference files (loaded in SKILL.md Steps 2-4) is always the foundation
- Live-fetched content augments and enriches on top of static knowledge
- Theme-based interleaved synthesis: weave sources together by topic, NOT source-by-source sections (per locked decision, consistent with Phase 1 inline citation style)
- When live-fetched info contradicts static knowledge, present both with dates: "As of [static date], X applied. However, [live source] from [date] indicates Y." Let the practitioner assess.
- No "Sources consulted" footer list -- inline citations are sufficient
- Response length: 400-800 words for multi-source synthesis (ceiling, not floor). If live content merely confirms static knowledge, a shorter response with a note that live sources confirmed current accuracy is appropriate
- Follow-up questions: content from WebFetch stays in Claude's context window. Do not re-fetch the same URLs within the same conversation

**Section 7: Degradation Handling** (~30 lines)
- Three tiers with specific language patterns:
  - Tier 1 (Partial failure -- some sources unavailable):
    > **Note:** [Source A] and [Source B] were unavailable during this research -- their coverage is based on built-in knowledge (last verified [date from file]).
  - Tier 2 (Total web failure -- all WebFetch attempts fail):
    > **Note:** Live source retrieval was unavailable for this query. This response draws from built-in reference knowledge. For the most current information, consult the sources cited directly, or retry this query later.
  - Tier 3 (Timeout/slow -- some sources still loading): provide answer from available sources + static knowledge, note which sources timed out, offer to retry
- Only mention failed sources that were selected as relevant to the specific query
- Known-blocked sources (FTC, EPIC, CDT) that are relevant: note them as "based on built-in knowledge" in degradation header, using "Last verified" dates from their nav guide files
- Degradation notes placed BEFORE the topic heading, not at the bottom
- Keep notes brief and factual, not apologetic -- practitioners routinely work with partial information

Total target: ~150-180 lines. Use horizontal rules between sections for visual separation (matching skill-behaviors.md pattern).
